Getting started

When you have read about the core principles underpinning the styleguide and the CSS implementation, you may want to check out the tutorial-style Getting started.

Core principles

The CSS is based on the principles behind OOCSS by Nicole Sullivan. It does not, however, use the OOCSS framework or any of its components. The following is an overview of the important principles that underly the implementation.

Separate components and layout

All layout concerns are handled by the layout system - individual components should not reimplement it. Components should focus strictly on adjusting positioning/spacing from the defaults made by the layout system, as well as the component "skinning" (backgrounds, borders, font adjustments and so on). This ensures that there is one authorative source for layout specification, enabling us to easily adjust if necessary, and keep centralized control over responsive layouts.

Components should always look the same

Any component should always look the same, regardless of its context. If you move a component from the left column to the right, it should still look the same. The layout details may give it different outer dimensions, but the internal skinning should not change. This ensures easily composable UIs and easy reuse of visual components.

Style classes, not elements

Styling directly on elements prevents multiple presentations of the same elements without resorting to overriding styles. Overriding styles lead to CSS bloat and is highly discouraged. Overriding styles will also interfere with the previous principle, and may cause surprising results when moving components around. Styling directly on classes avoids this problem. There are only a few cases where styles are applied directly to elements, and those situations are highlighted in this docment.

Avoid decendant selectors

Decendant selectors such as the child selector (p a) are generally discouraged because they may interfere with the principle that components should always look the same, regardless of their context. However, the closer together the elements in the selector are, the less likely it is that they will create a problem. (A selector as wide-reaching as .aside .media is strongly discouraged). Decendant selectors are used, but very sparingly.

Keyboard focus

All components should display focus outlines in a consistent way. Read more in the focus guide.

Grid layout

Page layout is controlled by two main classes, row and col. Additionally, there are classes to determine the width of individual columns, as well as classes to provide responsive overrides (discussed below). The layout is based on a grid of 12 columns, and you assign each column the desired amount of columns to span, e.g. class="col c3" for a column that spans three columns. Columns float from left to right, but can be made to float right with the use of the right class.

There is also a class .c-1of24 for making half-width columns (useful for button-only columns etc).

Headings

Headings are among the few tags that have styles applied directly to them. Headings are commonly used, so it is assumed that default styles (e.g. not having to class them all) will be useful. This is still currently undecided.

There are also classes .h1 etc to emulate the visual styles of the various headings, which can be used with any element (including headings - thus enabling semantic use of headers). These enable reusable components that can be shared with other projects that may not use the same baseline CSS. Reusable components should not make any assumptions about element styles - they should strictly rely on classes with defined behavior.

Icons

Icons are implemented as SVG. In order to use the icons, they must first be loaded on the page, then you can refer to them using xlink. To load the icons, place the following right after the opening body tag: 

<div id="icon-container" class="hide"></div>
          <script src="scripts/icons.js"></script>

The div will have all the available SVG icons loaded inside it, allowing you to refer to the icons later: 

<svg class="icon" aria-hidden="true" role="presentation">
          <use xlink:href="#icon-delete"/>
          </svg>

Some icons are designed to be displayed bigger than the default. Add the icon-l class to achieve this effect: 

<svg class="icon icon-l" aria-hidden="true" role="presentation">
          <use xlink:href="#icon-delete"/>
          </svg>

Icons can be highlighted with a blue color by adding the class icon-hl: 

<svg class="icon icon-hl" aria-hidden="true" role="presentation">
          <use xlink:href="#icon-delete"/>
          </svg>

Layout and spacing

Every block element (except un-classed divs) have default margins in all directions. This ensures consistent spacing between elements. Vertical margin collapse is leveraged to get spacing above a set of components, between each one, and again below the last one. Spacing may be overridden to achieve specific effects, usally by using the spacing-specific class names. To give a div element the same spacing as other block elements, give it the class mod.

Spacing classes

If you find yourself grabbing these often, it is probably a sign that the default spacing rules need to be refined. Spacing classes are made up from three letters: m (for margin), or p (for padding), followed by one letter for the direction (all, horizontal, vertical, top, right, bottom, left), followed by one letter for the size (none, small, medium, large).

Examples:

  • mtn (no top margin)
  • mvl (large top and bottom margins)

Vertical spacing for whole sections

The class tight-s ("tight, small") can be added to a block element, and will cause all block elements inside it to have smaller vertical margins than usual. You can also add htight-s or htight-xs for smaller horizontal margins.

Horizontal spacing for whole sections

When the default spacing is not enough, there are two more ways to group content: related and unrelated. You add the class name to a container element, and then the class item to every immediate child element you want to space out (usually all of them), and its horizontal margin will be adjusted accordingly. The media element is a concept from OOCSS. It is a generic structure that can be used to solve surprisingly many component-level layouts. It has a fixed-width "thing" (typically a picture, but can be anything) on the left and free-floating content on the right, which does not flow below the left-side content even if it is longer. Modal dialogs/glass panes

If you want to create the illusion of a modal dialog, you can place your dialog boxes (typically border-box, but really anything) in a glass pane. The glass pane overlays a semi-transparent dark layer on the whole page, and centers whatever content you put in it. Scripting

For predictive inputs to work without visual problems for all fonts, you need some JavaScript that copies the currently typed text from the .pi-input element to the pi-current-input element. Doing so, it is likely that certain letters will cover parts of the next letter in the prediction underneath. Unfortunately, the only solution to this is to manually retract a few pixels from the span's width on every keystroke. Not very elegant, but it works.

If you use the component with a font that does not have much "fuzziness"/anti-aliasing, you might get away without the span and the following script. Use at your discretion.

The script used on this page looks like: 

function mirror(el) {
  el.querySelector('.pi-input').addEventListener('input', function () {
    var content = el.querySelector('.pi-current-input');
    content.innerHTML = this.value;
    content.style.width = null; // Allow the browser to reassign automatic width
    content.style.width = content.clientWidth - 2 + 'px';
  });
}

var predictions = document.querySelectorAll('.predictive-input');

for (var i = 0; i < predictions.length; i++) {
  mirror(predictions[i]);
}

Responsive design

The approach to responsive design is "mobile first", or said differently: "small first". So default styles are intended for the smallest screens, then media queries are used to override these styles for bigger screens. There are currently five sizes: the default, small (> 640px wide screens) , medium (> 992px wide screens), large (> 1200px wide screens) and xlarge (> 1800px wide screens).

Some base styles vary between the various screen sizes - for example, h1 headings are smaller on the smallest screens.

Responsive layout

The layout can be individually specified for each of the three screen sizes. The default column-width classes (e.g. c3 and so on) apply to the smallest screens, and can be overridden for medium and large screens. To do so, use classes prefixed with rm- for medium screens, and rl- for large screens. The following example creates a column that is 12 columns wide on small screens, 6 on medium screens and 4 on large screens:

 

A varying sized column

When the 12 column grid is not sufficient to lay out a component, there are additional classes to set the width of columns: s1-5, which gives you a column one fifth (e.g. 20%) of the available width, and s1-7. They are also available with rm- and rl- prefixes, for responsive applications.

For places where space is a scarse resource (like the search results), add the expando-tight class to make the table display with less white-space.

Tooltips

These small tooltips can be used as an alternative to the built-in title attribute on links. This component only provides the proper look, positioning must be done manually. Note that the pointer must be included manually, as doing so offers some flexibility: you can omit it, you can include one that points upwards, or you can move it off center with JavaScript.

Tweening icons

Tweening icons can be used to transition from one icon to another. Currently this component is relatively specific to the X and + icons. Only one state is shown at a time. When transitioning, icons are rotated 45 degrees. ts1 ("tween state 1") starts at 0 degrees rotation, and rotates to 15 degrees as its opacity decreases to 0. ts2 starts at -45 degrees and rotates to 0 degrees as its opacity increases to 1.
